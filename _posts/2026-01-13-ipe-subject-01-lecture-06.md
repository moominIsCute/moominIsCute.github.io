## 객체지향 프로그래밍(OOP) 개요

**탄생**: 1966년 Simula67

**핵심 장점**
- 재사용성
- 유지보수성
- 확장성

### 기본 개념

| 용어 | 설명 | 예시 |
|------|------|------|
| **클래스(Class)** | 객체를 만들기 위한 설계도/틀 | 붕어빵 틀 |
| **객체(Object/Instance)** | 클래스로부터 생성된 실체 | 붕어빵 |
| **속성(Attribute)** | 객체의 상태/특성 | 스마트폰의 색상, 화면크기 |
| **메소드(Method)** | 객체의 동작/기능 | 전화걸기(), 문자보내기() |
| **메시지(Message)** | 객체 간 상호작용 방식 | 객체들은 메시지로 서로 대화 |

## 객체지향 4대 특성 ⭐️시험 필수

| 특성 | 설명 | 장점 | 예시 |
|------|------|------|------|
| **캡슐화<br>(Encapsulation)** | 데이터와 메소드를 하나로 묶고<br>내부 구현을 숨김 | • 보안성 향상<br>• 단순화 (내부 몰라도 사용 가능)<br>• 유지보수 용이 (내부 변경 시 외부 영향 없음) | 자동차 엔진: 내부 작동 원리를 몰라도 시동 가능 |
| **상속성<br>(Inheritance)** | 상위 클래스의 특성을<br>하위 클래스가 물려받음 | • 코드 재사용<br>• 일반화 → 특수화 구조 | 동물 → 강아지, 고양이 |
| **다형성<br>(Polymorphism)** | 같은 메시지에<br>다른 반응 | • 유연한 설계<br>• 확장성 향상 | 동물.소리() → 개는 "멍멍", 고양이는 "야옹" |
| **추상화<br>(Abstraction)** | 핵심 정보만 표현하고<br>불필요한 세부사항 제거 | • 복잡도 감소<br>• 필수 인터페이스만 제공 | 동물(추상) → 강아지(구체)<br>내부 로직은 숨기고 인터페이스만 노출 |

### 다형성의 핵심 기술

| 구분 | 오버로딩(Overloading) | 오버라이딩(Overriding) |
|------|----------------------|----------------------|
| **정의** | 같은 이름의 메소드를 여러 개 만드는 기술 | 부모 클래스의 메소드를 자식 클래스에서 재정의 |
| **조건** | 매개변수의 개수 또는 타입이 달라야 함 | 메소드 시그니처가 동일해야 함 |
| **발생 위치** | 같은 클래스 내 | 상속 관계의 클래스 간 |
| **예시** | `add(int a, int b)`, `add(int a, int b, int c)` | 부모의 `display()` → 자식에서 새롭게 정의 |

## 객체들의 관계 4가지

| 관계 유형 | 설명 | 예시 |
|----------|------|------|
| **1. 연관화(Association)** | 객체 간의 일반적인 관계 | 학생 ↔ 학교 |
| **2. 분류화(Classification)** | 클래스와 객체의 관계 | 김철수 → 사용자 클래스 |
| **3. 집단화(Aggregation)** | 부분-전체 관계 (약한 관계) | 배달기사 → 배달팀 |
| **4. 일반화(Generalization)** | 상속 관계 (is-a 관계) | 로켓배송 → 배송 |

## SOLID 원칙 ⭐️개발자 5대 덕목

객체지향 설계의 5가지 핵심 원칙

| 원칙 | 약어 | 설명 | 핵심 개념 |
|------|------|------|-----------|
| **단일 책임 원칙** | **S**RP<br>(Single Responsibility) | 한 클래스당 하나의 책임(기능)만 가짐 | 클래스는 변경 이유가 하나여야 함 |
| **개방-폐쇄 원칙** | **O**CP<br>(Open-Closed) ⭐️ | 확장에는 열려있고, 변경에는 닫혀있음 | 기능 추가 O, 기존 코드 수정 X |
| **리스코프 치환 원칙** | **L**SP<br>(Liskov Substitution) | 자식은 부모를 완벽하게 대체 가능 | 기능 일관성 유지 |
| **인터페이스 분리 원칙** | **I**SP<br>(Interface Segregation) | 인터페이스는 작고 명확하게 분리 | 불필요한 의존성 제거 |
| **의존성 역전 원칙** | **D**IP<br>(Dependency Inversion) | 구체 클래스가 아닌 추상 인터페이스에 의존 | 구체화가 아닌 추상화에 의존 |

**암기법**: **SOLID** (각 원칙의 영문 첫 글자)

## 디자인 패턴 3분류 ⭐️

### 1. 생성 패턴 (Creational Patterns)
**주제**: 객체를 어떻게 만들까?

| 패턴 | 설명 | 비유 |
|------|------|------|
| **싱글톤(Singleton)** | 단 하나의 인스턴스만 생성 | 유일무이한 객체 (대통령은 1명) |
| **팩토리 메소드(Factory Method)** | 조건에 따라 다른 객체 생성 | 객체 생성 공장 |
| **빌더(Builder)** | 단계별로 객체를 조립해서 생성 | 레고처럼 조립 |

### 2. 구조 패턴 (Structural Patterns)
**주제**: 어떻게 엮을까?

| 패턴 | 설명 | 비유 |
|------|------|------|
| **어댑터(Adapter)** | 호환되지 않는 인터페이스를 연결 | 만능 돼지코 (110V ↔ 220V) |
| **데코레이터(Decorator)** | 기능을 동적으로 추가 | 옷 겹쳐입기 |
| **파사드(Facade)** | 복잡한 시스템을 간단한 인터페이스로 제공 | 복잡함을 숨기고 간편하게 |

### 3. 행위 패턴 (Behavioral Patterns)
**주제**: 어떻게 협력할까?

| 패턴 | 설명 | 비유 |
|------|------|------|
| **옵저버(Observer)** | 상태 변화를 자동으로 통지 | 구독과 알림 (유튜브 알림) |
| **전략(Strategy)** | 알고리즘을 동적으로 교체 | 네비 경로 선택 (최단거리/고속도로 우선) |
| **커맨드(Command)** | 명령을 객체로 캡슐화하여 저장/취소 가능 | Ctrl+Z (실행 취소) |

- **O**: Open-Closed ⭐️(가장 중요)
- **L**: Liskov Substitution
- **I**: Interface Segregation
- **D**: Dependency Inversion

- **행**위 패턴: 동작/협력 관련
