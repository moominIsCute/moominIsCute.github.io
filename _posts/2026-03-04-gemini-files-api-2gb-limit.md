Files API를 쓰다 보면 누군가 한 번쯤은 "2GB 넘는 파일도 올릴 수 있지 않냐"는 질문을 한다. 문서에 비슷해 보이는 숫자가 두 개 나와서 혼동되기 쉽다. 어디서 오는 혼란인지 정리해봤다.

## 20GB와 2GB — 다른 얘기다

Files API 문서에는 두 가지 숫자가 나온다.

| 항목 | 한도 |
|------|------|
| **파일 하나당 크기** | 최대 **2GB** |
| **프로젝트 전체 저장 용량** | 최대 **20GB** |

20GB는 프로젝트 안에 올려둘 수 있는 **총합**이다. 파일 하나를 20GB까지 올릴 수 있다는 뜻이 아니다. 파일 하나는 반드시 **2GB 이하**여야 한다.

드라마 고화질 원본은 에피소드 하나가 3~5GB를 넘기도 한다. 그대로 올리면 업로드 자체가 실패한다.

## 그러면 어떻게 하나

방법은 하나다. **파일을 잘라서 올린다.**

영상을 일정 길이로 분할해 각 조각이 2GB 미만이 되도록 만들고, 조각마다 개별적으로 업로드한 뒤 Gemini에 순서대로 넘겨 분석한다. 이 방식을 **청크(chunk) 분할 업로드**라고 한다.

```
원본 영상 (4GB)
    ↓ ffmpeg으로 분할
chunk_01.mp4 (1.8GB)  →  Files API 업로드  →  Gemini 분석
chunk_02.mp4 (1.7GB)  →  Files API 업로드  →  Gemini 분석
    ↓
결과 합산
```

ffmpeg으로 분할하면 된다. 파일 크기 기준이 아니라 시간 기준으로 자르는 게 일반적이다. 화질과 원본 길이에 따라 적절한 구간을 정해두고 나누면 된다.

## 청크 분할의 한계

한 가지 문제가 있다. Gemini는 청크마다 **독립적으로** 분석한다. 앞 청크의 내용을 뒤 청크가 자동으로 기억하지 않는다.

청크 경계를 넘어가는 장면이 있다면, 그 장면의 맥락이 잘릴 수 있다. 이 문제는 청크 간 **오버랩 구간**을 두거나, 각 청크 분석 결과를 프롬프트에 함께 넣어주는 방식으로 보완할 수 있다.
